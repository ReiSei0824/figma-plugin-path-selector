# Figma插件路径规划师（Figma Plugin Path Selector）
## 核心定位

这个 skill 用来解决 2026 年 Figma 表面层快速变多后的一个真实问题：同一个设计需求，到底该做成 custom skills、generative plugins、code layers、传统 plugins，还是根本不该放在 Figma 里做。它先帮用户判断“值不值得做”，再判断“该落在哪个表面”，最后产出可以直接交给设计、产品、工程推进的交付 brief。

它不负责直接写完整插件代码，也不替代视觉设计本身；如果需求本质上只是一次性执行、没有稳定输入输出，应该明确建议“不做插件化”。

---

## 触发场景

- 用户说“这类设计流程该做成 Figma skill 还是 plugin”
- 用户已经有一段重复出现的设计协作流程，但不确定该放在 Figma 内还是放到外部工具/MCP
- 用户准备做生成式设计插件、设计团队 AI 工作流或模板化设计助手，需要先做路径选择和范围收敛
- 用户要写插件立项文档、需求说明或发布 brief，希望先把输入输出、权限边界和风控点讲清楚

---

## 工作流
### Step 1: 锁定目标工作流

先提取最小可重复单元：是谁在什么场景下反复做这件事、输入是什么、期望输出是什么、频率多高、当前最费时的步骤在哪。若用户描述过于抽象，强制追问到一个可以演示的具体任务样本；如果拿不出样本，就标记为“暂不适合插件化”。

### Step 2: 拆成动作与约束

把工作流拆成可判断的动作层：读设计稿、生成文案、批量改图层、排版约束检查、导出资产、跨工具取数、多人协同审阅。同步列出约束：是否要联网、是否需要读写本地文件、是否要访问团队私有数据、是否必须在画布内即时反馈、是否要跨页面/跨文件批处理。

### Step 3: 对照 Figma 表面层做路径比较

按下列顺序比较实现表面：
- `custom skills`：更适合提示驱动、范围可控、对设计上下文有依赖的团队内能力
- `generative plugins`：更适合一键生成多版本视觉输出、改图、变体探索
- `code layers`：更适合可复用的设计到代码桥接和带逻辑的界面片段
- `classic plugins`：更适合结构化批处理、图层操作、数据整理、批量规范化
- `outside Figma / MCP`：更适合跨系统审批、数据库取数、复杂后端逻辑

如果一个需求需要多种表面协同，必须明确主入口和次入口，不要默认“全都做”。

### Step 4: 评分并选 Top 1 路径

至少从五个维度评分：用户价值、实现复杂度、设计内闭环程度、数据/权限风险、两周内可落地性。若最高分方案仍然低于“可交付阈值”，直接给出“不建议做”的判断，并说明更轻的替代方案，例如模板、变量体系、人工 SOP 或外部自动化。

### Step 5: 产出可执行 brief

围绕 Top 1 路径输出一份可交付 brief：目标用户、触发动作、必需输入、系统上下文、输出格式、权限需求、失败兜底、观测指标、MVP 范围、以后再做的功能。若适合做发布物，顺带给出仓库名、README 结构和演示样例建议。

### Step 6: 做交付前自检

最后检查 brief 是否过度承诺、是否遗漏隐私/品牌资产边界、是否把“设计判断”错误外包给生成模型、是否误把一次性项目包装成长期产品。若存在这些问题，直接修订而不是把风险留给用户自己处理。

## Gotchas

1. **不要把“会不会做”误当成“该不该做”**
   很多需求技术上都能做，但如果使用频率低、输入不稳定、目标含糊，插件化只会制造维护成本。先判定重复性，再谈实现表面。

2. **不要默认生成式能力一定优于传统 plugin**
   如果任务核心是批量重命名、图层清洗、规范对齐，传统 plugin 往往更稳、更便宜、更好测。只有当“生成多版本结果”本身是核心价值时，才优先 generative plugin。

3. **不要把需要人类审美判断的最后一拍彻底外包给模型**
   例如品牌视觉、活动主 KV、重要营销页面。可以让模型给方向、变体、参考，但最终拍板标准要保留给人。

4. **不要忽略权限与资产边界**
   只要涉及品牌组件库、私有设计文件、客户素材、联网取数，就必须明确哪些数据能进模型、哪些只能本地处理、失败时如何降级。

5. **不要把跨系统问题硬塞回 Figma**
   如果任务实质上依赖审批流、表格、数据库、工单系统或 CMS，Figma 只适合作为前台入口，真正逻辑应落在外部服务或 MCP。

6. **不要同时立项多个入口**
   同一个需求若既想做 custom skill 又想做 plugin 又想做 code layer，通常说明范围没收敛。必须先定主路径，再把其余路径写成后续版本。
